# 🤖 Claude Code 15项技能速查手册

## 📱 技能1：移动端编程控制

### 使用方法

```bash
# 方式1：通过 CLAUDE.md 侧边栏
# 在 Claude Code 左侧切换到标签页，手机扫码配对

# 方式2：安装 mcp-notify
# 手机上给正在运行的会话推送消息、追加指令

# 方式3：CLAUDE_PORT
claude session attach    # 生成会话 + 二维码，手机扫码后控制
```

### 适用场景
- 通勤时让 CC 在家中的电脑上处理简单任务
- 长任务跑着，自己出门了，路上追加指令
- 查看进度，不打断主任务

### 真实案例
> Boris 在沙发上用手机让 CC 处理了一个简单任务，完成后手机收到通知

## 🔄 技能2：远程会话控制

### 使用方法

```bash
# 把云端会话拉回本地终端继续
claude session pull    # 单向：拉回本地

# 在公司电脑上启动会话 → 下班路上用手机批准权限请求 → 到家后在个人电脑继续
```

### 适用场景
- 公司电脑启动会话做重构 → 下班路上手机审批权限 → 到家后切到个人电脑继续
- 云端任务做到一半，想在本地终端接手
- 长任务中途需要审批危险命令，人不在电脑前

### 真实案例
> Boris 说他直接在设置里开了「默认始终开启 Pull」—— 会话不再绑定某台设备，跟着你走

## ⚡ 技能3：自动化循环（Loop + Perpetual Tasks）

### 使用方法

```bash
# Loop：当前会话内循环执行（退出即消失，适合短期监控）
# 例：每小时检查一次部署状态
/loop "检查部署，日志整理成列表发给我"

/# Perpetual：持久化定时任务（重启后依然在跑，适合长期自动化）
# 例：每天早上8点自动跑测试
/perpetual "每天8点自动跑测试，检查所有单元测试"

/# 权限请求路由到手机审批
/permission-remote-approve enable
```

### Loop vs Perpetual 对比

| | Loop | Perpetual |
|--|------|---------|
| 持久性 | 退出即消失 | 重启后依然在跑 |
| 时间 | 单会话内 | 定时任务 |
| 适用 | 一小时监控部署 | 每天早上跑测试 |
| 过期 | 1天自动过期 | 无 |

### 真实案例
> Boris 跑着一堆自动化循环，"分钟一次是什么概念？你不用再盯着，它会自动回复——这不是工具，是一个真正的同事在值班"

### 骚操作
```
把权限请求路由到远程 → 地铁里审批危险命令，不用等回到电脑前
```

## 🌐 技能4：生命周期钩子（Lifecycle Hooks）

### 使用方法

```javascript
// 在 CLAUDE.md 或项目 .claude/ 中配置
{
  "lifecycle": {
    "on-tool-call": {
      "before": "验证工具参数",
      "after": "自动运行测试"
    }
  }
}

// 工具名支持正则匹配
// 配置文件支持三个层级：全局 / 本地 / 项目
```

### 适用场景
```
写代码 → 自动跑单元测试 → 测试不过就自动修 → 整个链条不需要你盯着
```

### 核心类比
> "Git Hooks 是**提交后**的自动化；Lifecycle Hooks 是**代码生成时**的自动化"

## 👀 技能6：Browser Extension（最重要技巧）

### 使用方法

```bash
# 安装 Claude Browser Extension
# CC 会自主完成反馈循环：
# 写代码 → 自己打开浏览器 → 自己截图 → 自己读控制台报错 → 自己迭代修复

# 桌面端：开箱即用，内置浏览器
# 非桌面端：需要手动配置 Playwright
```

### 核心价值
> 作者以前是 CC 的「人类脚手架」：
> - CC 开发完 → 启动测试服务器 → 作者去浏览器看效果 → 截图发回去 → 复制控制台报错粘给它
> - **本质：作者在帮 CC 当它的眼睛**

### 升级后
> CC 自己看页面 → 自己截图 → 自己读报错 → 自己迭代修复
> 作者从「脚手架」变成「验收员」，只需最后看一眼结果

### Desktop 端优势
> 内置浏览器，开箱即用，比第三方方案更稳定可靠

## 🌳 技能7：会话分支（Session Branching）

### 使用方法

```bash
# 在长任务的分叉路口，创建分支
/branch                    # 在当前位置创建分支，两个会话共享之前上下文

# 两个分支各自往不同方向走，互不影响
```

### 适用场景

| 场景 | 操作 |
|------|------|
| 架构设计提了两个方案想都试试 | 直接 Branch，两个都跑 |
| 聊了半小时到分叉路口，想试不同方向 | Branch，不丢上下文 |
| 以前 | 只能复制粘贴开新会话，上下文全丢了 |

### 作者评价
> "这可能是15条里**最被低估**的功能"

## 👻 技能8：幽灵上下文（Ephemeral Context）

### 使用方法

```bash
# 在 CC 执行长任务时，临时问个快速问题
/ghost "这个函数的返回类型是什么？"
# 或
# 在 CC 工作时直接开口问，不打断主任务

# 幽灵特点：
# ✅ 只读模式，不触发工具，不修改文件
# ✅ 能看到当前对话全部上下文
# ✅ 不保存到主对话历史
# ✅ 关闭后答案消失
# ✅ 复用父会话，几乎无额外成本
```

### 适用场景

| 场景 | 以前 | 现在 |
|------|------|------|
| 长任务中途想问个函数返回类型 | 等做完/打断任务 | 直接 /ghost 问 |
| 突然想起来要查个东西 | 开新终端查 | 直接幽灵问 |
| 不确定某个 API 用法 | 停下手头工作查 | 边干边问 |

### Boris 原话
> Boris 说他同时跑着几十个 CC，靠的就是 Ghost——不污染主上下文，也不打断工作

## 🏗️ 技能9：Workspaces（沙箱隔离）

### 使用方法

```bash
# 在同一仓库中创建多个独立工作目录
/workspaces create --name 安全审查 --branch fix-security
/workspaces create --name 功能开发 --branch feature-new

# 各工作目录可以：
# ✅ 不同的分支
# ✅ 互不干扰的文件系统
# ✅ 自动创建独立沙箱
```

### 适用场景

| 场景 | 效果 |
|------|------|
| 同一仓库同时做不同任务 | 不会互相踩文件 |
| 写坏了也不影响主分支 | 安全隔离 |
| 并行跑多个 CC 干不同事 | 各走各的，不打架 |

### 作者评价
> "如果你在同一个仓库里同时跑多个 CC 做不同任务，它们会互相踩文件——Workspace 是一个独立的沙箱"

## 🚀 技能10：Mission Control（军团式编程）

### 使用方法

```bash
# 启动 Mission Control 编排
/mission-control

# CC 工作流程：
# 1. 先问"你想做什么"（访谈阶段）
# 2. 启动编排，自动调研影响范围
# 3. 拆分成 N 个独立执行单元
# 4. 每个单元在各自沙箱里并行跑
```

### 适用场景

| 场景 | 操作 |
|------|------|
| 大规模代码迁移 | 自动分析哪些文件可以并行改，同时开几十个各自干 |
| 可并行化的工作 | Mission Control 自动拆分 + 并行执行 |

### 真实案例
> 举例：把大型项目从 JavaScript 迁移到 TypeScript，以前只能一个个改 → 现在自动分批处理 + 几十个并行跑

### 作者震惊
> "这个功能我还没实际用过，但光想想就觉得恐怖——这不是辅助编程，是**军团式编程**"

## ⚡ 技能11：快速启动模式（Skip 自动发现）

### 使用方法

```bash
# 在 CLAUDE.md 或环境变量中配置
{
  "skip": {
    "auto-discovery": true,        # 跳过所有自动发现
    "auto-memory": true,            # 跳过自动记忆
    "tools": ["Read", "Edit", "Bash"]  # 只保留三个基础工具
  }
}

# 启动速度最高提升 10x
# 装越多 MCP Server，跳过越多，提速越明显
```

### ⚠️ 注意
> 钥匙串读取也被跳过了，需要通过环境变量或 CLAUDE.md 传入敏感信息

### 适用场景
- 写自动化脚本，不需要复杂上下文
- 大批量调用，每次省几百毫秒，累积下来很可观
- 裸配置用户也能受益

## 🔗 技能12：多仓库感知

### 使用方法

```bash
# 在一个仓库启动会话时，加入其他仓库
claude --repo /path/to/frontend --repo /path/to/backend

# CC 会：
# 1. 告诉你另一个仓库的存在
# 2. 给你读写权限
# 3. 加载 agents/ 目录下的规则（最新版本）
# 4. 跨仓库修改（改完库接口，自动去前端更新调用方式）
```

### 适用场景

| 场景 | 效果 |
|------|------|
| 前端项目依赖内部库 | CC 能同时理解两个仓库的代码 |
| 改了库接口 | 自动去前端项目里更新调用方式 |
| 跨仓库代码审查 | 同时审查多个仓库 |

### Boris 习惯
> 通常在一个仓库启动会话，然后通过指令把其他仓库加入会话

## 🎤 技能14：语音 + 打字混合输入

### 使用方法

```bash
# 混合输入模式：
# 1. 先打一半文字
# 2. 按住空格键
# 3. 说后半段
# 4. 松开空格，两种输入无缝拼接
```

### Boris 的用法
> **Boris 主要通过说话来写代码**
> 他大部分编码是「说话」而不是打字

### 适用场景
- 写长文案/代码，说比打快
- 突然有灵感，语音捕获比打字快
- 快速描述需求，不用逐字敲

### 未来趋势
> "编程的交互方式正从『打字』变成『对话』——以前你得准确描述每个技术细节，现在可以像和同事聊天一样说'把那个登录按钮颜色改成蓝色'"

### 使用方法

```bash
# 通过 mcp-notify，给正在运行的会话推送消息
# 不是开新会话，而是直接推送到已打开的会话

# 终端里跑着会话做重构
# 在手机上发："顺便把那个登录页也处理了"

```

### 适用场景

| 场景 | 效果 |
|------|------|
| 终端跑着重构 | 手机追加任务，不用停下手头工作 |
| 外部消息源接入 CC | iMessage、WhatsApp 等都可以桥接进来 |
| 持续运转的工作流 | 从任何地方发指令 |

### 核心意义
> "这和 Claude Code 的消息桥接机制异曲同工——让外部消息源接入 AI 产品，是必经之路"
